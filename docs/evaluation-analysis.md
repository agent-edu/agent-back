# 주식 에이전트 평가 — 규칙 기반에서 LLM-as-Judge까지

## 1. 개요

주식 전문가 AI 에이전트를 평가 데이터셋 35건으로 실험했다.
처음에는 규칙 기반 메트릭(ToolAccuracy, ResponseQuality)만으로 평가했고,
문제점을 발견한 후 GPT-4o를 판사로 활용하는 LLM-as-Judge를 도입했다.

- **에이전트 모델**: gpt-4.1-2025-04-14
- **판사 모델**: GPT-4o (temperature=0)
- **데이터셋**: `datasets/stock_agent_eval.csv` (35건, 8개 카테고리)
- **Opik 프로젝트**: doo-project
- **실험 스크립트**: `scripts/run_experiment.py`

---

## 2. 1차 실험 — 규칙 기반 메트릭만 사용

### 사용한 메트릭

| 메트릭 | 측정 방식 |
|--------|----------|
| **ToolAccuracy** | 기대 도구 vs 실제 호출 도구 문자열 비교 |
| **ResponseQuality** | 응답 길이 >= 20자면 1.0 |

### 결과

| 메트릭 | 점수 |
|--------|:---:|
| ToolAccuracy | **97.1%** |
| ResponseQuality | **100%** |

겉보기엔 거의 완벽하다. 하지만 실제 응답을 하나씩 보면 심각한 문제가 숨어 있었다.

---

## 3. 규칙 기반 메트릭이 놓친 문제들

### 3-1. 잘못된 회사를 조회했는데 만점

```
질문: "현대차 오늘 시세 알려줘"

에이전트 응답:
  현대차(종목 코드: 001500) 현재가: 11,220원 / 시가총액: 6,938억원

실제 문제:
  001500 = 현대비앤지스틸 (시가총액 ~7천억)
  005380 = 현대자동차 (시가총액 ~40조)
  → 완전히 다른 회사의 주가를 보여줌

ToolAccuracy: 1.0 ← 도구는 맞게 골랐으니 만점
ResponseQuality: 1.0 ← 20자 넘으니 만점
```

### 3-2. 조회 실패인데 만점

```
질문: "카카오 주가 얼마야?"

에이전트 응답:
  현재 카카오(016170)의 주가 정보를 조회할 수 없습니다.

실제 문제:
  016170 = 카카오 종목코드가 아님 (035720이 맞음)
  → 잘못된 종목코드로 조회 → 실패 → "조회할 수 없습니다"

ToolAccuracy: 1.0 ← 도구는 맞게 골랐으니 만점
ResponseQuality: 1.0 ← 82자니까 만점
```

### 3-3. "올해"라고 물었는데 2023년 데이터 반환

```
질문: "올해 상장한 기업 공시 알려줘"

에이전트 응답:
  2023년 1분기 동안 상장한 기업의 IPO 관련 공시는 다음과 같습니다...

실제 문제: 현재는 2026년인데 2023년 데이터를 반환
```

### 3-4. IPO 공시가 아닌 일반 공시 반환

```
질문: "최근 투자설명서 제출한 기업 알려줘"

에이전트 응답:
  1. 아바코 - 단일판매ㆍ공급계약체결
  2. 티이엠씨씨엔에스 - 금전대여결정

실제 문제: 투자설명서와 전혀 관련 없는 공시들
```

### 핵심

> ToolAccuracy는 **"어떤 도구를 골랐는가"**만 본다.
> ResponseQuality는 **"응답이 존재하는가"**만 본다.
> **"답변이 정확한가"는 아무도 보지 않는다.**

## 4. LLM-as-Judge 도입

### 개념

GPT-4o를 판사로 활용하여 에이전트 응답의 품질을 자동 채점한다.

```
[사용자 질문] → [주식 에이전트 답변] → [GPT-4o 판사가 1~5점 채점 + 사유]
```

### 평가 기준 (Rubric)

| 항목 | 기준 | 5점 | 1점 |
|------|------|-----|-----|
| **정확성** | 정보가 정확한가 | 핵심 데이터가 모두 정확 | 잘못된 정보, 조회 실패 |
| **완전성** | 질문 범위를 커버했는가 | 빠짐없이 답변 | 질문에 답하지 못함 |
| **유용성** | 유용한 형태로 정리했는가 | 구조화 우수, 투자 참고 가치 있음 | 유용하지 않음 |

### 구현

`scripts/run_experiment.py`에 3개 메트릭 클래스를 추가했다:

- `LLMJudgeAccuracy` — 정확성 채점 (0~1 정규화)
- `LLMJudgeCompleteness` — 완전성 채점
- `LLMJudgeHelpfulness` — 유용성 채점

3개 메트릭이 같은 판사 호출 결과를 공유하도록 캐시 처리하여, 35건 × 1회 = 35회만 GPT-4o를 호출한다.

## 5. 2차 실험 — LLM Judge 추가 결과

### Before vs After

| 메트릭 | 1차 (규칙 기반) | 2차 (LLM Judge 추가) |
|--------|:---:|:---:|
| ToolAccuracy | 97.1% | 97.1% |
| ResponseQuality | 100% | 100% |
| **LLM 정확성** | — | **71.4%** |
| **LLM 완전성** | — | **76.0%** |
| **LLM 유용성** | — | **74.9%** |

> **규칙 기반: "97~100% 완벽"**
> **LLM Judge: "실제 품질은 71~76%"**
>
> 이 25%p gap이 LLM-as-Judge의 가치다.

### 카테고리별 분석

| 카테고리 | 건수 | ToolAcc | Judge 정확성 | Judge 완전성 | Judge 유용성 |
|---------|:---:|:---:|:---:|:---:|:---:|
| 일반지식 | 5 | 100% | **96%** | 92% | 88% |
| 기업정보 | 5 | 80% | **92%** | 96% | 92% |
| 종합분석 | 2 | 100% | **90%** | 90% | 90% |
| 뉴스검색 | 5 | 100% | 72% | 76% | 76% |
| 주가조회_해외 | 5 | 100% | 60% | 80% | 80% |
| 주가조회_한국 | 5 | 100% | **52%** | 80% | 72% |
| IPO공시 | 5 | 100% | **48%** | 64% | 60% |
| 공모가 | 3 | 100% | **33%** | 47% | 47% |

### LLM Judge가 낮게 평가한 주요 케이스

| 질문 | ToolAcc | Judge 정확성 | 판사 사유 |
|------|:---:|:---:|------|
| 카카오뱅크 공모가 얼마였어? | 1.0 | **1/5** | 공모가 정보를 제공하지 못함 |
| LG에너지솔루션 공모가 정보 | 1.0 | **1/5** | 공모가 정보 완전 부재 |
| 카카오 주가 얼마야? | 1.0 | **2/5** | 종목코드 오류로 조회 실패, 사용자에게 떠넘김 |
| SK하이닉스 주가 조회해줘 | 1.0 | **2/5** | 주가 숫자 불일치 가능성 |
| 올해 상장한 기업 공시 | 1.0 | **2/5** | 2026년인데 2023년 데이터 반환 |
| 최근 IPO 공시 검색해줘 | 1.0 | **2/5** | IPO 무관 공시 혼재 |
| 현대차 오늘 시세 알려줘 | 1.0 | **3/5** | 종목코드 001500(현대비앤지스틸) 잘못 매칭 |

**모두 ToolAccuracy는 만점이지만 LLM Judge가 잡아냈다.**

## 6. 발견된 문제와 개선 방향

### 문제 분류

| 유형 | 건수 | 원인 | 개선 방향 |
|------|:---:|------|----------|
| 종목코드 매칭 오류 | 2건 | `_find_corp_code()` 부분 매칭 로직 | 줄임말 매핑 테이블 추가 |
| 조회 실패 | 2건 | 잘못된 종목코드 | 종목코드 검증 로직 |
| 시점 오류 | 1건 | "올해"를 날짜로 변환 못함 | 시스템 프롬프트에 현재 날짜 주입 |
| IPO 공시 관련성 | 2건 | DART API가 무관 공시 포함 | 도구에서 필터링 강화 |
| 공모가 데이터 부재 | 2건 | `get_ipo_price_info` 구조적 한계 | fallback 로직 개선 |
| 판사 보수적 채점 | 6건 | 도구 조회 데이터도 "검증 불가" 처리 | 판사 프롬프트에 "도구 데이터는 실시간" 지침 |

### 데이터셋 개선 포인트

| 문제 | 내용 |
|------|------|
| 카테고리 불균형 | 종합분석 2건, 공모가 3건으로 너무 적음 |
| 질문이 너무 직관적 | "삼성전자 주가 알려줘" → 에이전트가 틀릴 여지가 없음 |
| 라벨링 오류 | "네이버 홈페이지 주소" → 도구 없이 답해도 정답인데 ToolAcc=0.0 |

## 7. 주의 사항

1. **비용**: 35건 × GPT-4o 판사 = 소량이므로 무시 가능
2. **판사 일관성**: temperature=0으로 설정하여 점수 흔들림 최소화
3. **판사 편향**: 해외 주가에서 "정확성 검증 불가"로 보수적 채점 → 프롬프트 개선 필요
4. **순환 의존**: 판사도 LLM이므로 틀릴 수 있음 → 일부 샘플 수동 검증 권장
